# Render Deployment Guide

This guide explains how to deploy BookStore to Render with the repository Blueprint (`render.yaml`) and how to verify the deployment safely.

## Prerequisites

1. A GitHub repository containing this project.
2. A Render account with permission to create Blueprint resources.
3. GitHub Actions secret access if deploy hooks or Docker Hub publishing are required.

## Blueprint Resources

`render.yaml` provisions three resources:

| Resource | Render name | Runtime | Purpose |
| --- | --- | --- | --- |
| Database | `bookstore-db` | PostgreSQL | Production data store |
| Backend | `bookstore-api` | Docker | Spring Boot REST API |
| Frontend | `bookstore-web` | Docker | Next.js storefront/admin |

Both web services currently use `autoDeployTrigger: off`. This prevents accidental rebuilds and helps control free-tier pipeline minutes. Deploy manually from Render, or trigger a deploy hook only when the pipeline window is available.

## Step 1: Apply the Blueprint

1. Open [Render Dashboard](https://dashboard.render.com/).
2. Select **New +** -> **Blueprint**.
3. Connect the `Ecommerce_BookStore` repository.
4. Confirm that Render detects `render.yaml`.
5. Select the Singapore region if prompted.
6. Apply the Blueprint.

Render will create the PostgreSQL database and both Docker web services.

## Step 2: Confirm Backend Environment

For `bookstore-api`, confirm:

- `SPRING_PROFILES_ACTIVE=render`
- `DATABASE_URL` is bound from `bookstore-db`
- Fallback `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USERNAME`, and `DB_PASSWORD` are bound from `bookstore-db`
- `JWT_SECRET` is generated by Render
- `APP_BASE_URL` points to the frontend URL
- `BACKEND_PUBLIC_URL` points to the backend URL
- `APP_DEMO_SEED_ENABLED=true` only for the portfolio/demo deployment
- `GROK_ENABLED=false` unless a private `GROK_API_KEY` is configured

The backend health check path in `render.yaml` is:

```text
/api/health/live
```

For monitoring readiness after boot, use:

```text
/api/health/ready
```

## Step 3: Confirm Frontend Environment

For `bookstore-web`, confirm:

- `API_PROXY_TARGET` points to the backend `/api` URL
- `NEXT_PUBLIC_API_URL=/api`
- `NEXT_PUBLIC_BASE_URL` points to the frontend URL
- `NODE_ENV=production`
- `NEXT_TELEMETRY_DISABLED=1`

The frontend health check path in `render.yaml` is:

```text
/
```

The aggregate health endpoint is:

```text
/api/health
```

This endpoint checks frontend availability and backend reachability.

## Step 4: Optional Deploy Hooks

If GitHub Actions should trigger Render deploys:

1. Open the `bookstore-api` service in Render.
2. Copy the backend deploy hook URL.
3. In GitHub, open **Settings** -> **Secrets and variables** -> **Actions**.
4. Add `RENDER_DEPLOY_HOOK_BACKEND`.
5. Repeat for `bookstore-web` and add `RENDER_DEPLOY_HOOK_FRONTEND`.

Optional staging hooks for a `develop` flow:

- `RENDER_DEPLOY_HOOK_BACKEND_STAGING`
- `RENDER_DEPLOY_HOOK_FRONTEND_STAGING`

Important:

- Copy the frontend hook from the `bookstore-web` service, not from the backend, preview service, or an older frontend service.
- If a deploy hook was ever exposed, rotate it in Render and replace the GitHub secret.
- Do not repeatedly trigger hooks when Render reports that the workspace has run out of pipeline minutes. Wait for the next billing period or use a paid workspace.

## Registry Publishing

GitHub Actions publishes images to GHCR automatically. Docker Hub publish is enabled when these secrets exist:

- `DOCKERHUB_USERNAME`
- `DOCKERHUB_TOKEN`

If the repository variable `DOCKERHUB_NAMESPACE` is missing, the workflow uses `DOCKERHUB_USERNAME` as the namespace.

Semver-style registry tags:

- `latest`
- `v1.1.2`
- `v1`

Render Blueprint/source deployments still display commit hashes in Render deploy history. That is expected.

## Post-Deploy Verification

After both services deploy:

1. Backend live health
   - `https://bookstore-api-a1xl.onrender.com/api/health/live`
   - Expected: status `UP` or `ok`, depending on endpoint response shape.
2. Backend readiness
   - `https://bookstore-api-a1xl.onrender.com/api/health/ready`
   - Expected: service `UP`, database `UP`.
3. Frontend root
   - `https://bookstore-web-dr1k.onrender.com`
   - Expected: current BookStore storefront loads.
4. Frontend aggregate health
   - `https://bookstore-web-dr1k.onrender.com/api/health`
   - Expected: frontend `UP`, backend checks passing.
5. Smoke routes
   - `/products`
   - `/products/[id]`
   - `/flash-sale`
   - `/cart`
   - `/checkout`
   - `/login`

## Free-Tier Notes

- Free services can sleep after inactivity; first request after sleep can be slow.
- Demo data may top up in the background shortly after backend boot.
- Keep auto deploy disabled if pipeline minutes are limited.
- For a portfolio that must always be warm, use a paid Render plan or a controlled warm-up monitor.

## Troubleshooting

| Symptom | Likely cause | Action |
| --- | --- | --- |
| Deploy blocked before build | Render pipeline minutes exhausted | Stop retrying and resume when minutes reset |
| Backend health fails | Database env or profile mismatch | Check `SPRING_PROFILES_ACTIVE`, `DATABASE_URL`, and `DB_*` bindings |
| Frontend loads but API fails | Bad `API_PROXY_TARGET` | Confirm it points to backend `/api` |
| Frontend health cannot confirm commit | Wrong frontend deploy hook or stale service | Rotate/copy hook from `bookstore-web` |
| First page load is slow | Free service cold start | Wait for backend readiness and retry |
