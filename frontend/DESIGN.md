# Frontend Design Notes

This document maps the BookStore design system to the Next.js frontend implementation. The canonical product-level system lives in [../DESIGN.md](../DESIGN.md).

## Frontend Experience Goals

- Make the first screen feel like a real bookstore, not a landing-page mockup.
- Keep commerce actions obvious: browse, search, view detail, add to cart, checkout, and track orders.
- Keep mobile navigation, search, cart, language switching, flash sale, and chatbot usable without horizontal overflow.
- Keep admin screens compact, scannable, and operational.

## Route Expectations

| Route group | UX expectation |
| --- | --- |
| `/` | Strong storefront overview, featured books, promotions, no broken cover placeholders |
| `/products` | Searchable/catalog-like grid with stable product cards |
| `/products/[id]` | Clear book detail, price, stock/deal state, add-to-cart path |
| `/flash-sale` | Timed deal surface with active/upcoming states and professional sale hierarchy |
| `/promotions` | Coupon cards with readable terms and copy action |
| `/cart` | Responsive line items, totals, and login/checkout guidance |
| `/checkout` | Clear form, payment choice, totals, and error recovery |
| `/orders` | Authenticated order history with empty/loading states |
| `/admin/*` | Dense operations UI with tables, filters, badges, and responsive layout |

## Component Implementation Rules

### Header

- Desktop can show expanded navigation and actions.
- Mobile should keep only the strongest controls visible: brand, search, cart, language, and menu.
- Avoid wide account pills on mobile; account access belongs inside the menu.
- The flash-sale strip below the header must be compact and visually aligned with the current page.

### Product Images

- Use normalized image sources through the product image helpers.
- Public cards should not show placeholder covers unless no better fallback exists.
- Images must keep a stable aspect ratio so loading does not shift card height.

### Flash Sale Components

- `FlashSaleBanner` should read as a bookstore sale strip, not a generic ad banner.
- `FlashSaleCard` should emphasize cover, title, sale price, original price, discount, and progress.
- Time chips must preserve width across active/past/upcoming labels.
- Active state should be obvious without relying only on color.

### Chatbot

- Closed state should not leave a hidden panel that expands document width.
- Open state must fit inside mobile viewport.
- Copy should be short, helpful, and safe when the AI provider is disabled or degraded.
- Health should be driven by `/api/chatbot/health`.

### Forms

- Labels should stay visible or semantically available.
- Errors should sit near the field or action they explain.
- Avoid exposing raw backend/provider errors to the user.
- Loading buttons should keep the same width to avoid layout shift.

### Admin

- Use compact grids and tables.
- Prefer status badges, counters, and filters over large decorative cards.
- Admin mobile pages should stack cleanly and keep primary actions reachable.

## SEO and Metadata Rules

- Use route-specific metadata for public commerce pages.
- Titles should be absolute when generated by shared helpers to avoid duplicate brand suffixes.
- Keep Open Graph image at `/og-image.png`.
- Keep sitemap and robots routes valid after route changes.

## QA Checklist

Run these before considering the frontend ready for portfolio review:

```bash
npm run lint
npm run test:run
npm run build
BASE_URL=http://localhost:3001 npm run test:e2e:portfolio-audit
BASE_URL=http://localhost:3001 npm run test:e2e:journey
BASE_URL=http://localhost:3001 npm run test:e2e:admin-portfolio
npm run monitor:health
```

Manual visual checks:

- 390px mobile viewport has no horizontal scroll.
- Header actions do not overlap.
- Flash sale page uses the same palette as the rest of the storefront.
- Chatbot can open, send, close, and reopen.
- Product cards have completed covers.
- Checkout path is understandable with COD enabled.
- Admin dashboard is readable on desktop and mobile.
